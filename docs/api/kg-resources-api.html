    Resources

    Generic resources are rooted in the /v1/resources/{accountId}/{projId}/{schema_id} collection.

    Each resource…

    • belongs to a project ({projId})
    • inside an account ({accountId})
    • it is validated against a schema ({schema_id}).

    Create a resource using POST

    POST /v1/resources/{accountId}/{projId}/{schema_id}
  {...}
 ```

The json payload: 

- If the `@id` value is found on the payload, this @id will be used.
- If the `@id` value is not found on the payload, an @id will be generated as follows: `base:{UUID}`. The `base` is the `prefix` defined on the resource's project (`{projId}`).

## Create a resource using PUT
This alternative endpoint to create a resource is useful in case the json payload does not contain an `@id` but you want to specify one. The @id will be specified in the last segment of the endpoint URI.

    PUT /v1/resources/{accountId}/{projId}/{schema_id}/{resource_id} {…}


    The json payload. Note that if the payload contains an @id different from the `{resource_id}`, the request will fail.

## Update a resource

This operation overrides the payload.

In order to ensure a client does not perform any changes to a resource without having had seen the previous revision of
the resource, the last revision needs to be passed as a query parameter.

    PUT /v1/resources/{accountId}/{projId}/{schema_id}/{resource_id}?rev={previous_rev} {…}

    ... where `{previous_rev}` is the last known revision number for the resource.

## Tag a resource

Links a resource revision to a specific name. 

Tagging a resource is considered to be an update as well.

    PUT /v1/resources/{accountId}/{projId}/{schema_id}/{resource_id}/tags?rev={previous_rev} { “tag”: “{name}”, “rev”: {rev} }

    ... where 

- `{previous_rev}` is the last known revision number for the resource.
- `{name}` the name to give to the resource at a specific revision.
- `{rev}` the revision number to link the provided `{name}`.

## Deprecate a resource

Locks the resource, so no further operations can be performed. It also deletes the resource from listing/querying results.

Deprecating a resource is considered to be an update as well.

    DELETE /v1/resources/{accountId}/{projId}/{schema_id}/{resource_id}?rev={previous_rev}


    ... where `{previous_rev}` is the last known revision number for the resource.
